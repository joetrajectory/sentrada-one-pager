# Crossword layout engine

Generates a British-style crossword from a company's answer/clue candidates and
renders a print-ready A2 puzzle page onto a blank cream template, with real,
perfect text every time.

This is the third production format after the newspaper. It is a deliberate
sibling of `../newspaper/newspaper.py`: same Pango/Cairo text layout, the same
multiply-blend compositing so the paper texture shows through the ink, the same
resolution-independent zones (every position is a fraction of the template
dimensions, so one config works at any DPI), the same fontconfig font
registration, sRGB ICC tagging and print-production flags. Zero AI image-gen
dependency: the grid is generated by an algorithm and the page is typeset.

## Usage

```bash
python crossword.py --template crossword_template.png \
    --data test_cognism.json --output cognism_crossword.png

# print-ready A2 at 360 DPI (5953x8419), supersampled from the small template:
python crossword.py --template crossword_template.png --data test_cognism.json \
    --output cognism_crossword.png --render-dpi 360 --print-dpi 360

# reproducible grid (same seed -> same grid):
python crossword.py --template crossword_template.png --data test_cognism.json \
    --output out.png --seed 42

# validate without rendering a deliverable:
python crossword.py --template crossword_template.png --data test_cognism.json --check
```

The output PNG is written at the template's resolution unless `--print-dpi` /
`--print-size` downsamples it (and tags it with an sRGB profile).

## Setup

The engine uses Pango/Cairo for text layout and Pillow for compositing, exactly
like the newspaper engine. The grid generator (`grid_generator.py`) is pure
standard library.

```bash
../newspaper/setup.sh            # system libs (Pango, Cairo) + Python deps
# or, if the system libs are already present:
pip install -r requirements.txt
```

The fonts (Playfair Display, Lora) are the ones bundled in `../fonts`, registered
with fontconfig at runtime, so no system font install is needed. Override the
location with `SENTRADA_FONT_DIR`.

## The two images in this folder

| File | Role |
| --- | --- |
| `crossword_template.png` | The **blank** template the engine draws onto: clean cream paper and a warm dark-brown double-rule border. Same role as `../newspaper/newspaper_template_a2.png`. |
| `crossword_reference.png` | A **filled** reference showing the target layout, typography and feel. The engine's output is calibrated against it. |

Like the newspaper engine, every zone is a fraction of the image, so dropping in
a higher-resolution template (or a re-rendered one) needs no code change.

## Pieces

### `grid_generator.py`

Takes 25–30 candidate answer/clue pairs and places the best-fitting 15–20 into a
British-style (open) crossword:

- words cross one another and only the lettered cells exist (the paper shows
  through every gap) — **not** an American-style solid block with black squares;
- every word after the first intersects at least one already-placed word;
- no run-on words (the cells flanking a word along its axis stay empty) and no
  adjacent parallel words (a new letter never sits flush beside a parallel word);
- clue numbers are assigned left-to-right, top-to-bottom;
- many seeded attempts are run and the best (most words, then most compact and
  most-crossed) grid is kept.

Output (consumed by `crossword.py`): the 2D `grid` array, `rows`/`cols`, the
`numbers` map, the `placed` words (row/col/direction/number) and the `across` /
`down` clue lists.

Runnable standalone for debugging:

```bash
python grid_generator.py --data test_cognism.json --seed 42 --ascii
```

### `crossword.py`

Renders, top to bottom, onto the blank template:

- **Title** `THE [COMPANY] CROSSWORD` in Playfair Display Bold, centred, auto-sized
  to fit the page width (≈5% of page height).
- **Subtitle** the dry one-liner in Lora, centred, smaller.
- **Grid** the generated British-style grid: warm-white cells with thin near-black
  borders and small bold numbers, centred horizontally, ≈40–44% of page height.
- **Separator** a thin rule below the grid, spanning the content width.
- **Clues** two columns: `ACROSS` (left) and `DOWN` (right) in letterspaced
  spaced-caps (Playfair Display Bold), then numbered clues (numbers in Lora Bold,
  text in Lora regular, hung to a common indent). The clue size auto-fits so the
  taller column fills the available depth.
- **Credit** `sentrada` lowercase, small and discreet, bottom-right.

All text is warm dark brown (#3A2E22, matching the border); grid borders are
near-black (#1A1A1A); cell fills are a warm near-white. The ink text is multiplied
into the paper so it reads as printed; the grid (whose white cells must sit *on
top* of the paper) is composited over it.

## JSON schema (the Copy Agent contract)

`test_cognism.json` is a complete worked example.

| Field | Required | Notes |
| --- | --- | --- |
| `company_name` | yes | Used in the default title `THE [COMPANY] CROSSWORD`. |
| `title` | optional | Overrides the default title verbatim. |
| `subtitle` | optional | Defaults to "How well do you know your own company?" |
| `candidates` | yes | 25–30 `{answer, clue}` pairs (each may also carry `"anchor": true`). Answers are upper-cased and stripped to A–Z. The generator picks the best-fitting 15–20. |
| `anchors` | optional | List of raw answer strings to treat as heroes, as an alternative to the per-candidate `anchor` flag. |
| `min_words` / `max_words` | optional | Placement target (default 15 / 20). |
| `seed` | optional | Reproducible grid generation (CLI `--seed` overrides). |
| `grid_data` | optional | A pre-solved grid (the `grid_generator.py` output) to render verbatim, skipping generation. |

### Anchors (hero answers)

A candidate marked `"anchor": true` (or listed in the top-level `anchors` array) is
a designated hero the generator places first and tries hardest to land, so the
concept the brief is built around anchors the grid rather than merely surviving as
a clue. The longest anchor seeds the grid. If a designated anchor cannot be placed,
`--check` reports it as a FAIL (so the runner halts) rather than quietly dropping it.

## CLI flags

Matches the newspaper engine: `--template`, `--data`, `--output`, `--check`,
`--render-dpi`, `--print-dpi`, `--print-size`, `--render-size`, `--seed`,
`--ink-blur`, `--bleed-mm`, `--crop-marks`.

## Safety guards

Like the newspaper engine, the render refuses to ship a page it cannot stand
behind: required fonts must resolve to their real faces (no silent substitution),
and any layout failure (clues that will not fit, a grid overrunning the separator,
a non-viable grid) writes a `*.FAILED.png` for inspection and leaves the
deliverable path empty, exiting non-zero so a batch runner treats it as an error.

**Solvability.** `--check` re-derives the across/down runs from the final grid
geometry and asserts the puzzle is actually solvable, as a FAIL: every numbered run
matches its placed answer (length and letters, so crossings resolve), there are no
orphan runs or unchecked letters, and clue numbers and grid runs are in bijection.
The generator places by real intersection so this holds by construction, but the
assertion guards against any future change to placement or numbering silently
shipping a grid that breaks in the recipient's hands.
